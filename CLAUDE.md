# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A formal verification benchmark for [curve25519-dalek](https://github.com/dalek-cryptography/curve25519-dalek), a Rust elliptic curve cryptography library. Rust code is extracted to Lean 4 via [Aeneas](https://github.com/AeneasVerif/aeneas), then spec theorems are proved. The `dataset.jsonl` contains benchmark entries — theorems with `sorry` placeholders that agents must replace with correct proofs.

## Key commands

```bash
# Build the entire Lean project
lake build

# Build a single module (faster for iteration)
lake build Curve25519Dalek.Specs.Field

# Sync status tracking after any spec change
lake exe syncstatus   # updates functions.json and status.csv

# Run the agent evaluation harness
python scripts/eval_claude_code.py
python scripts/eval_claude_code.py --limit 3              # smoke test
python scripts/eval_claude_code.py --ids to_bytes_spec_074536e
python scripts/eval_claude_code.py --output results/run1.jsonl --resume
python scripts/eval_claude_code.py --dry-run --limit 3    # no claude calls

# View evaluation results
python scripts/show_results.py results/claude-code-*.jsonl
```

Requires `ANTHROPIC_API_KEY` and `claude` CLI on PATH for evaluation runs.

## Architecture

### Lean project layout

- `Curve25519Dalek/` — main verification library
  - `Types.lean`, `Funs.lean` — **auto-generated by Aeneas, never edit manually**
  - `Specs/` — hand-written spec theorems mirroring the Rust module structure
- `Utils/` — shared proof utilities
- `Curve25519Dalek.lean`, `Utils.lean` — module root files
- `lakefile.toml` — Lean package config; `lean-toolchain` pins `v4.28.0-rc1`

### Evaluation pipeline

Each dataset entry (`dataset.jsonl`) records a theorem introduced at a specific git commit. The harness (`scripts/eval_claude_code.py`):
1. Creates an isolated git worktree checked out at `commit_before`
2. Symlinks a shared Lake package cache (`~/.cache/dalek-lake-packages/`) to avoid rebuilding deps
3. Spawns `claude --print --bare` with a prompt asking the agent to fill the `sorry`
4. Grades pass/fail: `lake build <module>` exits 0 **and** no `sorry` remains in the file
5. Per-entry budget: $3 USD, 600s timeout

### Spec theorem style

New specs use Aeneas WP (weakest precondition) style:

```lean
@[progress]
theorem fun_name_spec
    (a : ArgType)
    (h : precondition) :
    fun_name a ⦃ result : ResultType =>
      postcondition_1 ∧
      postcondition_2 ⦄ := by
  sorry
```

`⦃` / `⦄` are typed as `\{{` / `\}}` in VS Code. Older specs use existential form (`∃ result, f = ok result ∧ ...`) — both are acceptable but new specs should use WP style.

## Code style rules (from CONTRIBUTING.md)

- Follow Mathlib4 naming and style conventions
- **No flexible tactics**: replace `simp` with `simp only [...]` (use `simp?` to discover the list), same for `simp_all`. Prefer `omega`/`ring`/`linear_combination` over `grind`
- **`maxHeartbeats` increases**: use multiples of 200000; refactor into helper lemmas if budget is very large
- Resolve linter warnings; do not suppress them
- After editing any spec file, run `lake exe syncstatus` and commit the updated `functions.json` and `status.csv`

## Files never to edit

- `Curve25519Dalek/Types.lean` — auto-generated by Aeneas from Rust types
- `Curve25519Dalek/Funs.lean` — auto-generated by Aeneas from Rust functions

To fix extraction output, modify `aeneas-tweaks.txt` or the upstream Rust source, then re-run extraction.
